VENTYLAB - GUÍA DE USO DE CURSOR Y CLAUDE CODE

Esta guía especifica cómo usar herramientas de IA para implementar VentyLab.

================================================================================
1. HERRAMIENTAS DISPONIBLES
================================================================================

A. CURSOR FREE
   Límite: ~500 completions/mes
   Modelo: GPT-4o (gratis limitado)
   Usar para: Refactors pequeños, correcciones, componentes simples

B. CURSOR PRO
   Límite: Ilimitado GPT-4o, ~500 usos de Claude Sonnet
   Modelo recomendado: Claude Sonnet 3.5
   Usar para: Implementación de módulos completos, lógica compleja

C. CLAUDE CODE
   Límite: Según plan de Anthropic
   Modelo: Claude Sonnet 3.5 o Opus
   Usar para: Arquitectura crítica, módulos de tiempo real, integración MQTT

RECOMENDACIÓN GENERAL:
- Refactorización: CURSOR PRO con Claude Sonnet
- Módulo Simulation: CLAUDE CODE
- Módulos Teaching/Evaluation/Admin: CURSOR PRO
- Componentes UI simples: CURSOR FREE

================================================================================
2. ESTRUCTURA DE PROMPTS
================================================================================

FORMATO OBLIGATORIO:

Implementar cliente MQTT para comunicación con ventilador.

- Lee ARCHITECTURE.md sección Simulation

- Telemetría: ventilab/device/001/telemetry
- Comandos: ventilab/device/001/command

- QoS: 1 (at least once)

- Reintentos automáticos: 5
- Delay exponencial: 1s, 2s, 4s, 8s, 16s
- Loggear intentos de reconexión

Manejo de errores:
- Connection refused → loggear y reintentar
- Message parsing error → loggear y continuar
- Publish error → loggear y propagar

Verificación:
- Se conecta correctamente
- Recibe mensajes del topic
- Publica mensajes correctamente
- Reconecta automáticamente

Implementar HexParser para parsear trama hexadecimal del ventilador.

Contexto:
- Lee CONTRACTS.md sección Trama Hexadecimal

Ubicación: ventylab-server/src/modules/simulation/hex-parser.ts

Estructura de trama:
[START] [TYPE] [LENGTH] [DATA...] [CHECKSUM]

Tipos de mensaje:
- 0xA1: Presión
- 0xA2: Flujo
- 0xA3: Volumen
- 0xA4: Alarma

Responsabilidades:
- Validar estructura de trama
- Verificar checksum
- Parsear según tipo
- Retornar objeto tipado
- Manejar tramas inválidas

Método principal:

Validaciones:
- Verificar START byte
- Verificar longitud
- Verificar checksum
- Validar tipo conocido

Manejo de errores:
- Trama inválida → throw InvalidFrameError
- Tipo desconocido → throw UnknownTypeError
- Checksum incorrecto → throw ChecksumError

Logging:
- Loggear tramas inválidas
- NO loggear cada trama válida (demasiado volumen)

Verificación:
- Parsea correctamente cada tipo
- Detecta tramas inválidas
- Verifica checksum
- Retorna tipos correctos

EJEMPLO 1 - Implementar SimulationService

Herramienta: CLAUDE CODE

Implementar SimulationService en ventylab-server/src/modules/simulation/simulation.service.ts

Contexto:
- Lee CONTRACTS.md sección 3.1 Simulation Backend
- Lee ARCHITECTURE.md sección 5 Módulo Simulation

Responsabilidades:
- Orquestar comunicación entre MQTT y WebSocket
- Validar comandos antes de enviar
- Parsear datos recibidos
- Broadcast a clientes conectados

Dependencias a inyectar:
- ventilatorConnection: IVentilatorConnection
- simulationGateway: ISimulationGateway
- hexEncoder: HexEncoder
- hexParser: HexParser

Métodos principales:
1. sendVentilatorCommand(command: VentilatorCommand): Promise<void>
   - Validar comando (rangos seguros)
   - Encodear a hex
   - Publicar vía MQTT
   - Loggear comando enviado

2. handleVentilatorData(hexData: Buffer): void
   - Parsear hex a objeto
   - Crear VentilatorReading
   - Broadcast vía WebSocket
   - Loggear si hay errores

3. initialize(): Promise<void>
   - Conectar MQTT
   - Suscribirse a telemetría
   - Setup handlers

4. cleanup(): Promise<void>
   - Desconectar MQTT
   - Limpiar subscripciones

Validaciones en sendVentilatorCommand:
- PEEP: 0-20 cmH₂O
- FiO₂: 0.21-1.0
- Volumen tidal: 200-800 ml
- Frecuencia: 5-40 resp/min

Manejo de errores:
- Usar Result<T, E> pattern
- Loggear con contexto
- NO crash por errores de parsing

Logging:
- Nivel INFO: Comandos enviados, conexiones
- Nivel ERROR: Fallos de validación, errores de parsing
- Nivel DEBUG: Cada dato recibido (opcional)

Verificación:
- Implementa correctamente cada método
- Valida parámetros médicos
- Maneja errores sin crash
- Logs apropiados

EJEMPLO 2 - Implementar useVentilatorData

Herramienta: CURSOR PRO

Implementar hook useVentilatorData en ventilab-web/src/features/simulator/useVentilatorData.ts

Contexto:
- Lee CONTRACTS.md sección 3.1 Simulation Frontend
- Lee ARCHITECTURE.md sección 5.2 Separación Lógica vs Visual

Return type:
UseVentilatorDataReturn

Responsabilidades:
- Escuchar evento ventilator:data del socket
- Acumular últimos 300 readings
- Proveer último reading
- Estado de conexión
- Manejo de errores

Usar:
- useSocket() de shared/hooks/useSocket
- useState para buffer de datos
- useEffect para suscripción
- useRef si es necesario evitar re-renders

NO hacer:
- Cálculos matemáticos (eso es para useChartCalculations)
- Llamadas API directas
- Lógica compleja

Patrón:
export function useVentilatorData(): UseVentilatorDataReturn {
  const { socket, isConnected } = useSocket();
  const [data, setData] = useState<VentilatorReading[]>([]);
  const [error, setError] = useState<string | null>(null);

  useEffect(() => {
    if (!socket) return;

    const handleData = (reading: VentilatorReading) => {
      setData(prev => [...prev.slice(-299), reading]);
    };

    const handleError = (err: any) => {
      setError(err.message);
    };

    socket.on('ventilator:data', handleData);
    socket.on('ventilator:error', handleError);

    return () => {
      socket.off('ventilator:data', handleData);
      socket.off('ventilator:error', handleError);
    };
  }, [socket]);

  const latest = data[data.length - 1] || null;

  return { data, latest, isConnected, error };
}

Verificación:
- Retorna exactamente UseVentilatorDataReturn
- Limita buffer a 300 elementos
- Cleanup correcto en useEffect
- Maneja errores apropiadamente

7. DEBUGGING CON CURSOR
================================================================================

Si el código generado no funciona:

1. Verificar que cumple contrato
   - Revisar CONTRACTS.md
   - Comparar shape de retorno

2. Verificar ubicación
   - Revisar FOLDER_STRUCTURE.md
   - Confirmar imports correctos

3. Verificar lógica
   - Revisar ARCHITECTURE.md
   - Confirmar que sigue patrón de capas

4. Regenerar con prompt más específico
   - Incluir el error encontrado
   - Pedir corrección específica

Prompt de debugging:
El código generado tiene el siguiente error: [ERROR]

Archivo: [RUTA]
Contrato esperado: [CONTRATO]

Corregir manteniendo:
- Ubicación del archivo
- Cumplimiento de contrato
- Arquitectura de capas
